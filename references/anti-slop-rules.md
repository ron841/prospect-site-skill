# Anti-Slop Rules Reference

Part of the prospect-site skill v0.7. Loaded by Phase 4 during design engine decisions, Phase 5 during generation, and Phase 6 during final verification. This file is the no-fly zone. Everything in it is a pattern the skill must NOT produce.

The reason this file exists is v0.6. In v0.6, the skill generated four different "prospect sites" that all looked identical. Same centered hero. Same three-column feature grid. Same testimonial carousel. Same rounded-corner buttons in the same blue. The businesses were an electrician, a plumber, a pressure washer, and a pool builder, and they all looked like the same Tailwind template with different logos pasted in. That is slop, and slop kills the sale. A contractor who opens their preview and recognizes the template instead of recognizing their own business will not sign the agreement.

The v0.6 failure had four root causes:

1. **No explicit "do not use this" list.** The skill had positive instructions ("use Features 8") but no negative instructions. Without a no-fly zone, the LLM defaults to the most statistically common pattern in its training data, which is the exact pattern every other AI site uses.
2. **No color discipline.** Colors were picked by vibe rather than by examining the actual logo. Every site drifted toward the same safe navy-and-white palette.
3. **No typography rules beyond "use a nice font."** The LLM defaulted to Inter or Poppins every time.
4. **No copy discipline beyond voice rules.** Plant Street voice handled the body copy well but let generic headline patterns like "Your Local [Trade] Experts" slip through.

This file fixes all four root causes with explicit banned lists.

---

## How the skill applies these rules

Anti-slop rules are applied at three points in the workflow:

**Phase 4 (design engine).** Before the skill presents the design proposal to Ron for approval, the chosen font pair, color palette, and hero mode are checked against the banned lists in this file. If any element violates the no-fly zone, the skill selects an alternative before showing the proposal. If no valid alternative exists (rare), the skill flags the conflict to Ron explicitly in the approval gate rather than silently picking something bad.

**Phase 5 (generation).** During HTML and copy generation, banned copy patterns, banned component structures, banned photography choices, and banned interaction patterns are filtered out. Content that matches a banned pattern is rewritten before being written to the output file.

**Phase 6 (verification).** A final anti-slop sweep runs over every generated page. This is a hard gate. Any slop pattern that slipped past Phase 4 or Phase 5 causes Phase 6 to fail the build. Fail fast, fail visibly. Do not ship slop.

## Banned fonts (by tier)

The wrong font choice is the fastest way to make a site look AI-generated. These rules define what the skill can and cannot use per tier.

### Globally banned as primary fonts for Premium and Professional tiers

These fonts are the AI default signal. Every LLM that has been asked to "make a nice website" has been trained on examples that used these fonts. Using them for Premium or Professional tier is the same as putting a "generated by AI" watermark in the footer.

- **Inter as a heading font** — the ChatGPT default, the Vercel default, the modern SaaS default. Instantly reads as AI-generated when it appears as the primary heading typeface for Premium or Professional tier. Inter as a body font is fine (it was designed for screen reading and is genuinely excellent) and is in fact the universal body font for all three tiers in v0.7. The slop signal is Inter-as-heading paired with other slop markers, not Inter per se.
- **Roboto** — the Google Material Design default. Generic and dated.
- **Poppins** — the "hip startup" font. Overused to the point of being meaningless.
- **Open Sans** — the WordPress default. Genericness peak.
- **Helvetica Neue / Helvetica** — too safe, reads as corporate clipart.
- **Montserrat** — the 2015-2020 "trendy" font that is now the 2026 "outdated" font.
- **Lato** — same problem as Montserrat.
- **Nunito** — the "friendly" font that every Canva template uses. Instant template read.
- **Source Sans Pro** — Adobe default, no character.
- **Raleway** — elegant in 2014, now reads as someone who stopped paying attention to typography in 2014.

### Tier-specific allowed fonts

For **Premium tier (score 70-100)**, the skill uses one of:

- **Fraunces** (primary heading, default) — a variable serif with real character and warmth. Distinctive without being weird.
- **Playfair Display** (heading) — classic high-contrast serif. Use when Fraunces does not match the vertical (e.g., rustic trades).
- **DM Serif Display** (heading) — slightly more geometric serif for modern trades.
- **Instrument Serif** (heading) — elegant display serif for premium design/build clients.
- **Space Grotesk** (body, paired with serif heading) — modern sans with character, not boring.
- **Manrope** (body) — clean geometric sans. Less common than Inter.
- **IBM Plex Sans** (body) — distinctive sans with personality, especially for technical trades.

For **Professional tier (score 40-69)**:

- **Space Grotesk** (primary, default) — modern, clean, not Inter.
- **DM Sans** (body) — clean and neutral without being generic.
- **Manrope** (body alternative).
- **Outfit** (heading or body) — geometric sans with warmth.
- **Figtree** (body) — newer font, reads modern without being trendy.

For **Standard tier (score 0-39)**:

- **Inter for both heading and body**, with a system font stack as fallback: `"Inter", -apple-system, BlinkMacSystemFont, "Segoe UI", sans-serif`. At Standard, the tier signal is simplicity and speed, not a distinctive heading. Using one font for both heading and body is consistent with the minimal aesthetic of the tier. The system font fallback handles load instability without changing the intended design.
- **The body font is universally Inter across all three tiers.** What changes by tier is the heading font, not the body font. Standard is the only tier where the heading font is also Inter (because Standard's tier signal is minimalism, not heading distinctiveness).

### The serif-heading discipline

Premium tier sites use a serif heading font paired with a sans-serif body font. This is the single strongest signal that a site is not AI-generated, because AI defaults almost always pick a sans-serif for everything. The serif-heading pair is visible in the first 100 milliseconds of a site visit and sets the quality expectation for everything that follows.

Phase 4 enforces this: if the selected tier is Premium, the heading font must be a serif from the allowed list. If Phase 4 somehow selects a sans-serif heading for a Premium tier site, the check fails and Phase 4 picks Fraunces as the fallback.

### Font loading discipline

Beyond the choice of font, the loading strategy matters for Lighthouse scores and visual stability. Phase 5 enforces:

- Fonts loaded via `@font-face` with `font-display: swap` to prevent invisible text during load
- Maximum two font families per site (one heading, one body). Three or more font families is slop.
- Maximum three weights per family (e.g., 400, 500, 700). Loading a full 100-900 weight range is wasteful and slow.
- Self-hosted fonts via `woff2` files in `assets/fonts/` preferred over Google Fonts CDN for privacy and performance
- If Google Fonts CDN is used, the `<link>` uses `display=swap` in the query string

## Banned color combinations

Color discipline was the biggest single failure mode in v0.6. The Johnson Brothers miss (deep navy and amber when the real brand was bright royal blue and sun yellow) was the clearest example. Colors must come from the actual logo, not from vibe-matching the business category.

### Globally banned color patterns

These are banned regardless of tier and regardless of what the logo looks like:

- **Pure black (`#000000`) + pure white (`#FFFFFF`) as the entire palette.** Flat and lifeless. Real sites use off-blacks (`#0B0B0D`, `#1A1A1A`) and off-whites (`#FAFAFA`, `#F5F5F5`) for depth.
- **Gradient backgrounds on hero sections.** Purple-to-blue, blue-to-teal, pink-to-orange gradients are the universal signal for "AI-generated SaaS landing page." Banned for contractor sites at all tiers. The v0.7 hero architecture is the Glassmorphism Trust Hero glass card on real prospect photography, or a solid brand color for the zero-photo fallback. There is no legitimate use case for a gradient background in this architecture. "Subtle" and "desaturated" gradients are banned too; fuzzy exceptions become loopholes every time.
- **Neon-on-dark color schemes.** A plumber's site should not look like Cyberpunk 2077. Saturated neon against black is a design-portfolio vibe and is wrong for home services.
- **All-monochrome grayscale palettes.** Reads as a developer portfolio, not a business. Real businesses have brand colors.
- **The "Florida Gator" combo: deep navy + amber/orange when the logo does not support it.** Named after the Johnson Brothers miss. Even if the business is in Florida, this specific combo is only used if the logo actually uses these colors.
- **Primary blue (`#0070F3` Vercel blue, `#1E88E5` Material blue, `#3B82F6` Tailwind blue-500).** These specific hex values are the instantly recognizable "generated by a framework default" signals. Use a color extracted from the actual logo instead.
- **Tailwind default color palette applied directly.** If the site's CSS contains `bg-blue-500`, `bg-gray-900`, `text-gray-600` as visible design choices (not as underlying utility classes), the palette is Tailwind default and reads as such.

### Color extraction discipline

Phase 4 pulls brand colors from the actual logo file using `node-vibrant` per the `image-handling.md` spec. The extracted palette is mapped to semantic roles:

- `Vibrant` → primary brand color
- `DarkVibrant` → primary hover state, dark backgrounds
- `LightVibrant` → light background tints, highlights
- `DarkMuted` → secondary text, muted UI elements
- `Muted` → subtle borders, dividers
- `LightMuted` → page background tints

If the extracted `Vibrant` color has low saturation (below ~40%), the logo is monochromatic or grayscale. In that case:

1. Check if the site CSS has a theme-color meta tag or brand color variable
2. Check if Instagram photos have a dominant color that is brand-consistent
3. Fall back to a neutral palette: warm off-black + warm off-white + one accent color borrowed from the trade category (e.g., warm red for electricians, deep blue for plumbers, forest green for landscapers)

If the extracted `Vibrant` color is very close to banned defaults (Tailwind blue-500, Vercel blue, Material blue), shift it slightly off — either toward a warmer tone or a darker shade — so the final site does not read as framework default.

### Banned secondary color choices

Even when the primary color is pulled correctly from the logo, secondary color choices can still produce slop:

- **Accent colors that don't exist in the logo.** If a plumber's logo is blue and white, the accent color should be a complementary tone from the blue family or a warm neutral, not a random teal pulled from nowhere.
- **Too many accent colors.** More than three visible colors on a contractor site reads as unfocused. The palette should be: primary brand + dark background + light background + one accent + neutral text.
- **Color-on-color stacking with insufficient contrast.** Light blue text on medium blue background, dark purple on dark blue. Phase 5 must verify WCAG AA contrast (4.5:1 for body text, 3:1 for large text) on every color combination in the generated site.

## Banned layout patterns

Layout is the second biggest slop signal after typography. These patterns are banned regardless of tier.

### The "SaaS landing page" hero

Banned pattern:

- Centered headline
- Centered subheadline
- Two buttons below the subheadline ("Get Started" / "Learn More")
- Dashboard mockup or gradient background
- No prospect-specific visual anchoring

Every AI-generated site defaults to this pattern. It signals "template" instantly. The v0.7 hero pattern uses the Glassmorphism Trust Hero glass card with asymmetric positioning, prospect-specific background imagery, and fact-dense copy that could not belong to any other business.

### Identical-card feature grids

Banned pattern: three or four services displayed as identical cards in a symmetrical grid. Same size, same padding, same icon treatment, same text length.

The v0.7 pattern is the asymmetric Bento services grid with one featured card (2x2 size with a photo) and 4-6 standard cards. The asymmetry is what prevents the grid from reading as a template. A perfectly symmetrical grid of identical cards reads as "ChatGPT made this."

### Centered-everything homepage

Banned pattern: every section on the homepage has centered headings, centered text, and centered buttons. This is the "default Tailwind" layout signal.

The v0.7 pattern mixes alignment: left-aligned body text for readability, asymmetric section layouts, occasional right-aligned accents for visual rhythm. Centered alignment is used sparingly and intentionally (e.g., in the stats section headline), not as a default.

### The "trust badge" logo strip without real badges

Banned pattern: a horizontal strip of grayscale logos labeled "As featured in" or "Trusted by" when the business has no actual press coverage or partnership logos.

The v0.7 rule: trust badges are only rendered if Phase 2 captured real credentials (license numbers, BBB accreditation, trade association memberships, manufacturer certifications). Fabricating trust badges is a cardinal sin.

### Full-width video backgrounds in the hero

Banned for v0.7. Video backgrounds are often used to signal production quality but introduce three problems: they tank Lighthouse scores, they fail on mobile data connections, and they are almost never the prospect's own footage (meaning they are generic stock clips of hands on keyboards or drone shots of suburbs).

Hero video is allowed ONLY if the prospect provided their own video content, which is rare for Marion County contractors. If Phase 2 captures no real video assets, no video hero.

### Symmetric testimonial carousels

Banned pattern: four identical testimonial cards in a row, each with a circular avatar, identical star rating, and quote text of similar length. Reads as LinkedIn connections list.

The v0.7 pattern for testimonials varies card sizes based on quote length, uses real Google review formatting (five stars, reviewer first name, date, Google logo), and mixes in the actual review count and rating header above the cards. The visual variety is what makes it feel real.

### Auto-playing sliders and carousels

Banned globally. Auto-advancing content that moves on its own is a usability disaster, an accessibility failure, and a visual signal of template slop. Users need to be in control of their own reading pace.

Any testimonial or image section that rotates content must:

1. Be manually advanced (user clicks or swipes)
2. Have visible pagination dots or arrows
3. Pause on hover or focus
4. Not rotate on page load

### Modal popups on page load

Banned globally. No "sign up for our newsletter" popup, no "we use cookies" modal on load (cookie banner is a non-blocking footer strip), no email capture overlay, no "exit intent" popup when the user's cursor approaches the back button.

The v0.7 rule: the user's first experience with the site is the site itself, not a popup blocking the site. Aggressive conversion tactics are wrong for home services where trust is the primary purchase driver.

### The "stats on a white background" pattern

Banned pattern: a section with three or four centered numbers (e.g., "50 Years", "500 Projects", "4.9 Rating") on a plain white background with small labels underneath.

The v0.7 pattern for stats is a full-width dark section with oversized numbers (8-12rem font size), white or brand-accent text, and a visible break in the page rhythm from the surrounding light sections. Dark stats sections are mandatory per SKILL.md, not optional.

### Horizontal scrolling sections

Banned globally. Horizontal scrolling on a vertical page is confusing, breaks keyboard navigation, and fails on mobile. If content does not fit in a vertical layout, split it into multiple sections or use a progressive disclosure pattern (expandable cards).

## Banned component patterns

### Buttons labeled "Learn More"

Banned globally. Already enforced by the voice rules in `content-rules.md` and the internal linking rules in `seo-geo.md`, but repeated here because it is the single most common slop pattern.

Every button says something specific about what happens next:

- `Call (352) 465-4600` (not "Contact Us")
- `Request a quote` (not "Learn More")
- `See all electrical services` (not "View Services")
- `Read Sarah's full review` (not "Read More")

### "Scroll down to see more" indicators

Banned globally. The bouncing arrow, the pulsing circle, the "Scroll" label at the bottom of the hero — all of these assume the user does not know how to scroll. It is condescending, visually noisy, and a template signal.

### Floating chatbot widgets

Banned globally for v0.7. Chatbots on contractor sites are cart-blanche slop. The contractor is the chatbot; the prospect should call the contractor. Adding a chatbot to a contractor site is a template signal and dilutes the conversion path.

No exceptions in v0.7. If a specific client raises the question in the future, it becomes a separate project conversation that is not baselined in this skill.

### Sticky header that occupies more than 80px on mobile

Banned pattern: a sticky header that takes 100-150px of vertical mobile screen real estate, pushing the content down and eating the hero visibility.

The v0.7 rule: sticky header on mobile is under 80px tall, contains only the logo and a tel: link (or hamburger menu), and does not duplicate the full nav that appears lower on the page.

### Forms with more than six fields

Banned globally. Every additional form field reduces conversion. Contractor leads need: name, phone, email (optional), service (dropdown), message, submit. That is five visible fields plus the submit button. Anything more is friction.

The honeypot hidden field does not count toward the six-field limit.

### Cookie banners that block content

Banned pattern: a full-screen or fixed-bottom cookie banner that blocks scrolling or covers more than 15% of the viewport. These banners exist primarily for GDPR compliance and the v0.7 target market (Marion County home services customers) does not require them.

If the client specifically requests a cookie banner, use a minimal implementation: a thin bottom strip with "Got it" acknowledgment that is dismissible, does not block scrolling, and disappears after first interaction. No multi-option consent forms.

## Banned copy patterns

These are beyond the voice rules in `content-rules.md`. Voice rules handle the big bans (passionate about, committed to excellence, leverage, stakeholders). This section handles the slop patterns that can slip past voice rules because they use technically allowed words.

### Banned headline patterns

- **"Your [trade] experts"** — meaningless, generic, every contractor site uses it. Example ban: "Your Ocala Plumbing Experts." Even with a real city name, the pattern is slop.
- **"Quality [trade] services"** — zero facts. Ban.
- **"Serving [area] since [year]"** as a standalone headline — fine as a subheadline fact but too thin as a headline. Ban unless combined with identity or action: "Serving Ocala since 1975. Call (352) 465-4600." is acceptable.
- **"Welcome to [business name]"** — the WordPress default. Ban.
- **"About Us"** as a page heading — allowed as nav label only. The actual page h1 should say something like "Fifty years of electrical work in Marion County." Ban the generic "About Us" as h1.
- **Any headline starting with "We" or "Our"** — headlines should focus on the prospect's facts (years, reviews, location, credentials), not on generic first-person claims. Ban "We provide quality service." Allowed: "50 years of quality service in Dunnellon" where the years and city are real facts.
- **Headlines ending in exclamation points** — banned per voice rules but repeated here. "Call Now!" is slop.
- **Headlines using all caps** — banned. "CALL (352) 465-4600" in an h1 is shouty slop. Use normal case and let typography weight carry emphasis.

### Banned body copy patterns

- **"Get a free quote today"** — replace with a specific action. "Call (352) 465-4600 for a free quote" is acceptable because it names the specific action.
- **"Don't hesitate to contact us"** — the single most generic call-to-action in contractor-site history. Ban. Replace with "Call" or "Text" plus the actual phone number.
- **"Proudly serving [area]"** — fine once as a subheadline, banned as repeated filler.
- **"Fully licensed and insured"** without the actual license number — ban. The license number is the citable fact. Use "Florida license EC13007855" instead of "fully licensed."
- **"Family-owned and operated"** without naming the family — ban. Use "Owned by Ben and Wendy Tucci" with real names instead.
- **"Years of experience"** without a specific number — ban. Use "50 years of experience" with a real number.
- **"Customer satisfaction is our top priority"** — cliché. Ban.
- **"No job too big or too small"** — cliché. Ban.
- **"We treat your home like our own"** — cliché. Ban.

### The "could apply to any business" test

The single best slop detector for copy is the "could this apply to any business in any trade in any city" test. Read any paragraph of generated copy and ask: if I replace the business name with another contractor's name, does the paragraph still make sense?

If yes, the paragraph is slop and must be rewritten with specific facts that only apply to this one business.

Example failed test:

> Our team is dedicated to providing the highest quality service to every customer. We take pride in our work and go the extra mile to ensure your complete satisfaction.

This paragraph could apply to a plumber, an electrician, a law firm, a dentist, or a landscaping company. It is slop. Rewrite:

> Ben Tucci has wired houses in Dunnellon for 50 years. His father wired them for 40 years before that. If your panel was installed before 2005, Ben has probably worked on it.

Specific, grounded, impossible to apply to any other business.

## Banned photography and image choices

### Banned stock photography

These are hard bans globally:

- **Generic handshake photos.** Universal slop signal. Nobody's business is "shaking hands."
- **Photos of smiling teams in matching polo shirts** when the team is not the actual prospect's team. These are stock photos of models, and they read as stock immediately.
- **"Business person pointing at a whiteboard" stock photos** — universal LinkedIn post slop.
- **Photos of tools arranged on a white background** — reads as a Home Depot product page, not a contractor.
- **AI-generated people photos** — banned unconditionally. Stable Diffusion faces, Midjourney people, or any synthesized human images. If a photo has that "uncanny valley" feel, it is banned.
- **Stock photos of the wrong trade.** Landscaping photos on an electrician site. Plumbing photos on a roofing site. Phase 2 must verify the category match before Phase 5 uses an image.
- **Stock photos of the wrong geography.** Snow-covered mountains on a Florida site. Desert cacti on a Marion County site. The `image-handling.md` rules require geographic category matching for stock fallbacks.
- **Photos with visible watermarks.** Watermarked stock = unpaid stock = instantly unprofessional.

### Banned photo composition patterns

- **Every photo on the site being a wide-angle exterior shot.** Variety is essential.
- **Every photo being a portrait-orientation shot.** Mix landscape and portrait based on the layout slot.
- **Every photo being the same brightness and color temperature.** A photo set that has been over-processed to look uniform reads as stock or AI-enhanced.
- **Low-resolution photos stretched beyond their native size.** Phase 5 must not upscale a 400px photo to a 1200px slot.
- **Photos that contain the contractor's logo as a watermark** — fine on social media, wrong on their own website.

### The real-photo-first rule

Per `image-handling.md`, the photo priority order is:

1. Google Business Profile photos (via Google Places API)
2. Photos scraped from the prospect's own website
3. Instagram photos (if prospect has a public account)
4. Social Images folder (Ron's curated Marion County photography)
5. Curated stock (Unsplash, Pexels, with strict category and geography filters)

The real photos always beat the stock photos, even when the real photos are lower quality. A slightly blurry photo of the actual business truck in front of the actual Dunnellon address is worth ten perfect stock photos of generic electrical panels.

## Banned interaction patterns

### Aggressive parallax scroll

Banned pattern: hero backgrounds that move at dramatically different speeds than the foreground, creating a nauseating depth illusion. This was fashionable in 2014 and has not aged well.

Subtle parallax (hero background moves at ~50% of scroll speed) is allowed in Premium tier only. Aggressive parallax (elements flying past at different directions, section-by-section depth changes, 3D tilt on scroll) is banned at all tiers.

### Hover effects that shift layout

Banned pattern: cards that grow when hovered, pushing adjacent content sideways. This is disorienting and accessibility-hostile. Hover effects that are allowed: subtle shadow increase, color shift, subtle underline animation. Anything that changes the layout position of content is banned.

### Scroll-jacking

Banned globally. Taking over the user's scroll behavior to force them through a specific sequence of content reveals is hostile and unnecessary. The user scrolls at their own pace.

### Custom cursors

Banned globally at all tiers. No exceptions. Custom cursors are a vanity feature that fails on four dimensions: they break accessibility (users with motor disabilities rely on the OS default cursor for accurate targeting), they break user expectation (desktop users have decades of muscle memory with the native cursor), they add JavaScript weight and CSS complexity for zero conversion benefit, and they read as "tried too hard" rather than polished. Even on Premium tier design/build clients where the case for cursor polish is strongest, the right move is to invest that polish budget in typography, spacing, color, and photography, not in cursor gimmicks. Default OS cursors only.

### Click sounds or audio feedback

Banned globally. Contractor sites should not make sounds on interaction. The user's phone may be on silent or the user may be in a public place.

### Unsolicited chatbot popups

Already covered under banned components but repeated here: no automated chat widgets that pop up with "Hi! Can I help you find something?" after X seconds. This is the universal template slop signal.

## Banned mobile patterns

The SKILL.md six-check self-test requires Ron to open every preview on his phone and run an eye test. Mobile is not a reduced version of the desktop experience; it is the primary experience for contractor prospects (who find contractors via phone searches while standing in their flooded basement). Mobile slop is therefore first-class slop.

### Tap targets smaller than 44x44px

Banned globally. Apple HIG specifies 44x44 points as the minimum tap target for reliable finger accuracy. Buttons, nav links, phone-link icons, and form submit buttons all need at least 44x44px of clickable area (including padding, not just visible content).

Phase 5 must verify minimum tap target sizes on every interactive element. Phase 6 check 15 flags any tap target under 44x44.

### Body text smaller than 16px on mobile

Banned globally. iOS Safari auto-zooms input fields when the font size is under 16px, which causes a jarring viewport shift on form interaction. Beyond the iOS issue, anything under 16px is hard to read on a phone and reads as "template not designed for mobile."

The v0.7 rule: body text is 16px minimum on mobile, 17-18px preferred. Headings scale up from there. No text anywhere on the site is under 14px, and 14px is reserved for fine print in the footer only (copyright, credits).

### Hamburger menus that hide the phone number

Banned globally. This is explicitly called out in `section-patterns.md` but belongs here as an anti-slop rule too. A contractor site where the phone number is tucked inside the hamburger menu on mobile is broken. The phone number must be visible in the mobile sticky header at all times, as a tap-to-call button.

The v0.7 pattern: mobile sticky header contains logo on the left, tel: button with phone icon on the right, and a hamburger in the far-right corner for the nav. The phone button is the first thing a prospect sees when they land on the site on their phone, because that is the action we want them to take.

### Bottom CTA bars that overlap the iOS home indicator

Banned pattern: a fixed bottom CTA bar that sits flush against the bottom of the viewport on iPhones, overlapping the home indicator (the horizontal bar at the bottom of iOS devices without a home button). This causes accidental dismissal of the CTA when users swipe up from the bottom.

The v0.7 rule: every fixed bottom element uses `padding-bottom: env(safe-area-inset-bottom)` or similar CSS to respect the iOS safe area. Phase 6 check 15 verifies safe area handling on bottom-positioned fixed elements.

### Unintentional horizontal scroll on mobile

Banned globally. If the mobile site has any horizontal scroll (triggered by an oversized image, an unconstrained table, an overflowing form field, or a layout that doesn't respect `max-width: 100vw`), it is broken and slop.

Phase 6 check 15 tests every generated page at 375px viewport width (iPhone SE baseline) and verifies `document.body.scrollWidth <= 375`. Any page that scrolls horizontally fails the build.

### Phone numbers rendered as plain text instead of tel: links

Banned globally. Every phone number on every page must be wrapped in an `<a href="tel:+1XXXXXXXXXX">` link so mobile users can tap-to-call. Phone numbers as plain text (no link) or as spans with onClick handlers are broken.

The v0.7 rule: Phase 5 wraps every phone number occurrence in a tel: link with the E.164 format phone number in the href and the display format in the link text. Phase 6 check 15 verifies every visible phone number in the HTML is inside a tel: anchor.

### Forms without autocomplete hints

Banned globally. Mobile form fields should have appropriate `autocomplete` attributes so the user's saved contact info pre-fills: `autocomplete="name"`, `autocomplete="tel"`, `autocomplete="email"`. Missing autocomplete is a minor slop signal but a real conversion leak on mobile.

### Modals that don't close with tap-outside or swipe-down

Banned globally. Any modal that opens on mobile (e.g., the mobile nav drawer, an image lightbox) must be dismissible by tapping outside the modal content or swiping down. Modals that require finding and tapping a specific close button are mobile-hostile.

## Banned technical patterns

### Client-side JavaScript frameworks

Banned in v0.7 contractor sites. React, Vue, Svelte, Next.js, SvelteKit, Nuxt — all overkill for a five-page contractor site. The skill generates plain HTML, CSS, and minimal vanilla JavaScript. This is a deliberate v0.7 constraint per `SKILL.md` and `deployment.md`.

Why v0.7 bans frameworks:

- Frameworks add build complexity, deploy complexity, and bundle size for zero user benefit on sites this small
- Plain HTML ships faster, loads faster, and scores higher on Lighthouse (no framework runtime to parse and execute)
- Claude Code can edit plain HTML instantly; framework projects require understanding component files, state management, routing, and build pipelines
- No framework means no build step means no CI/CD required for updates, which keeps client update turnaround under 24 hours (per `deployment.md` update workflow)

**This is a v0.7 constraint, not a permanent architectural rejection.** At v0.9+ when clients start requesting interactive features (booking calendars, quote calculators, multi-step estimate forms, project galleries with filtering), the right answer will usually be a vanilla JS component for that specific feature, not introducing a framework for the whole site. But if a specific client need justifies a framework (e.g., a franchise client with 20 locations needing dynamic routing), that becomes a separate conversation at that point, not a v0.7 baseline decision.

For v0.7: plain HTML only, no exceptions.

### Heavy JavaScript animation libraries

Banned: GSAP, ScrollMagic, Lottie, Three.js, Rive. These are powerful tools for the right project, but wrong for contractor sites. They add weight and complexity without matching the conversion goal.

Allowed: minimal vanilla JS for the mobile menu toggle, the before/after slider interaction, the FAQ accordion expansion. Nothing beyond that.

### CSS preprocessors

Banned: Sass, Less, Stylus in the generated output. The skill may use preprocessors internally during development, but the output shipped to Vercel is plain CSS.

### Multiple analytics scripts

Banned: loading Google Analytics AND Plausible AND Fathom AND Mixpanel on the same site. Pick one analytics platform (Plausible recommended per `deployment.md`) and stick with it.

### Tracking pixels without user consent

Banned: Facebook Pixel, TikTok Pixel, Google Ads conversion pixels loading without an explicit client decision to use them. These require consent in some jurisdictions and add weight without matching the home services conversion path.

## Icon discipline

### Banned icon patterns

- **Emoji as icons in navigation or service cards.** 🔧 next to "Repairs" is not an icon, it is an emoji, and different operating systems render emoji differently, producing inconsistent visual output.
- **Font Awesome classic style icons** (the 2012-era stroke icons). Dated.
- **Material Design icons pasted without adjustment.** Recognizable as Material immediately.
- **Mixed icon styles.** Using Lucide for some icons and Heroicons for others on the same site is slop.
- **Inconsistent icon sizes or weights.** Every icon in a section should be the same size, same stroke weight, same visual rhythm.

### Allowed icon sources

The v0.7 skill uses **Lucide** icons exclusively (via `lucide-static` or inline SVG). Reasons:

- Consistent stroke-based style across the full library
- Modern (2020+) design that has not aged into slop
- Accessible SVG output
- Free and open source
- Comprehensive coverage for home services (tools, trucks, houses, phones)

Icon matching rules:

- Each service in the services grid gets a specific Lucide icon matched to the service type
- All icons on a page use the same stroke width (typically 1.5 or 2)
- All icons on a page use the same size (typically 24px or 32px)
- Icon color inherits from the surrounding text color for consistency

## Banned date and placeholder patterns

This category was underweighted in the original draft and matters because date and placeholder slop are the telltale signs of an abandoned or AI-generated site. A prospect who sees "© 2019" in the footer or "[BUSINESS NAME]" somewhere in the copy immediately loses trust.

### Stale copyright years

Banned pattern: a footer that reads "© 2019 T&F Electric" on a site built in 2026. The copyright year must reflect the current year at generation time, and the quarterly content refresh protocol in `deployment.md` includes a copyright year verification step every 13 weeks.

The v0.7 rule: Phase 5 writes `&copy; [current-year] [Business Name]. All rights reserved.` where `[current-year]` is dynamically resolved to the actual year at generation time, not hardcoded. Phase 6 check 16 verifies the footer copyright year equals the current year.

### Placeholder text in production output

Banned unconditionally. Any instance of `[BUSINESS NAME]`, `[PHONE]`, `[ADDRESS]`, `example.com`, `(555) 555-5555`, `123 Main Street`, `Lorem ipsum`, `foo@bar.com`, `John Doe`, `Jane Smith`, or any other template placeholder in the rendered HTML is a hard build failure. Phase 6 check 16 greps the output for these patterns and fails the build on any match.

This is already enforced by `seo-geo.md` check 11 (schema validation requires no placeholders) but repeated here because placeholder leaks are the worst kind of slop. A site with placeholder text in production is worse than no site at all.

### Stale "last updated" or "posted" dates

Banned pattern: any visible "Last updated: April 2024" on a site deployed in April 2026. If the generated site has a "last updated" element anywhere (footer, about page, blog section), it must either reflect the actual build date or be removed.

The v0.7 rule: the skill does not generate "last updated" visible elements unless Phase 2 explicitly captured an update cadence the prospect wants displayed. If in doubt, omit the element. Stale dates are worse than no dates.

### "Since [wrong year]" language

Banned pattern: "Since 2024" or "Serving Ocala for 10 years" when the real founding year is different or the years-in-business math is wrong. Every year-based claim must tie to the real founding year in `profile.json`.

The v0.7 rule: Phase 5 computes years-in-business from `profile.json.business.foundingDate` and the current year, never hardcoded. Phase 6 check 16 cross-checks every "since [year]" and "[N] years" claim against the captured founding year.

### Forward-dated content

Banned pattern: "Coming in 2027" or "Expanding services in 2028" or any forward-dated promise the prospect did not explicitly make. These signals look like outdated site content that has drifted past its own deadline.

The v0.7 rule: no forward-dated content unless Phase 2 captured a specific future-dated claim from the prospect's own site (e.g., "Grand Opening July 2026" would be allowed if the prospect's site says exactly that). Otherwise, omit.

### Stale dateModified in schema

Banned pattern: the LocalBusiness or Service schema includes a `dateModified` field from a prior build that was not updated on the current rebuild. AI crawlers use `dateModified` as a freshness signal; a stale value tells Perplexity and ChatGPT that the page has not been maintained.

The v0.7 rule: Phase 5.5 writes `dateModified` as the current build timestamp in ISO 8601 format on every schema block that includes it. Phase 6 check 16 verifies the `dateModified` value is within 24 hours of the current build time.

### Copyright or credit lines that reference the builder

Banned pattern: footer lines like "Site designed by [agency name]" or "Powered by GRM Web Services" or similar. These are builder-credit lines, and they belong on the builder's portfolio, not on the client's live site.

The v0.7 rule: generated footers never include GRM attribution. The site belongs to the client, and the footer reflects only the client's business. GRM's portfolio work happens on GRM's own site at getrootedmedia.com, not through credit links on client sites.

## The "eye test" for slop detection

When running the final self-check in Phase 6, use this rapid slop-detection heuristic to supplement the explicit bans above.

### The five-second test

Look at the homepage for exactly five seconds. Without reading the copy, answer:

1. **Does this look like a template?** If yes, it is slop. The immediate visual impression should be specific enough that you could not copy the layout to another business without obvious fit problems.
2. **Does this look AI-generated?** If yes, it is slop. The markers are: Inter typography, blue-and-white palette, symmetric layouts, gradient heroes, identical cards, centered everything.
3. **Could I tell what trade this is without reading the text?** If no, the imagery and color choices are wrong.
4. **Could I tell what city this is without reading the text?** If no, the location signaling is too weak. Should include a real skyline, real street, or real local landmark in imagery.
5. **Does anything on the page feel "off" in a way I can't articulate?** That vague discomfort is often the clearest slop signal. Investigate what is producing the feeling.

### The Ron eye test

From `SKILL.md` self-check item 6: Ron opens the URL on his phone and would text it to a family member without qualifying or apologizing.

If Ron's gut reaction would be "this is close but I'd have to explain it," the site is slop and must be fixed before deployment. "Close enough" is not the standard. "Ron would proudly send this to his sister without a caveat" is the standard.

### The swap test

Swap the business name for another business in the same trade. Does the site still look fine? If yes, the site is generic and has slop. A non-slop site is specific enough that the swap would immediately produce visible fit problems (wrong owner names, wrong location photos, wrong colors, wrong credentials).

## Common mistakes to avoid

- **Do not treat anti-slop rules as optional suggestions.** Every rule in this file is a hard constraint. Phase 4, Phase 5, and Phase 6 enforce them as hard gates.
- **Do not assume the default LLM choice is fine.** The default LLM choice is almost always slop because training data overweights common patterns. Always check the default against this file's banned lists and swap if needed.
- **Do not ship a Premium tier site with Inter as the heading font.** The serif-heading rule is non-negotiable for Premium. Inter as the body font at Premium is correct and expected (per css-framework.md) because the body font is universally Inter across all tiers; the Premium tier signal lives in the Fraunces heading, not in the body text.
- **Do not ship a site with gradient hero backgrounds.** The SaaS gradient is the single strongest slop signal.
- **Do not fabricate trust badges, credentials, or testimonials to fill empty space.** An empty section is better than a slop section.
- **Do not use stock photos of people who are not the real team.** Real photos always beat stock, even at lower quality.
- **Do not let voice rules pass on headlines that technically comply but feel generic.** "Your Ocala Plumbing Experts" passes the voice ban list but fails the swap test. Ban it on the swap-test failure, not on a specific banned phrase.
- **Do not add chatbots, modal popups, or aggressive conversion widgets to contractor sites.** These are wrong for the market and read as template slop.
- **Do not mix icon styles.** Lucide-only rule is non-negotiable.
- **Do not ship the five-second test failure.** If the homepage reads as AI-generated in five seconds, fix the fonts, colors, or layout before deploying.

## Self-check before Phase 6 marks anti-slop verification complete

Phase 6 anti-slop verification runs as check 15 (final sweep) after all other checks pass. Each check item is labeled `[auto]` for items Claude Code can verify programmatically, or `[human]` for items that require Ron's visual judgment on the preview URL.

The `[auto]` items run during Phase 6 as automated hard gates. The `[human]` items run as part of the six-check self-test after Phase 7 deploys the preview, and before Ron sends the URL to a prospect.

### Automated checks (Phase 6 hard gates)

**Typography:**

- [ ] `[auto]` Primary heading font is not on the banned list for the assigned tier
- [ ] `[auto]` Body font is not on the banned list for the assigned tier
- [ ] `[auto]` If Premium tier, heading font is a serif from the allowed list
- [ ] `[auto]` No more than two font families loaded
- [ ] `[auto]` No more than three weights per family

**Color and visuals:**

- [ ] `[auto]` Color palette was extracted from the real logo via node-vibrant (verified by design decision log or profile.json extraction method field)
- [ ] `[auto]` Hero section does not use a CSS gradient background (grep for `linear-gradient`, `radial-gradient` in hero styles)
- [ ] `[auto]` No banned color combinations present (Tailwind blue-500, Vercel blue, Material blue-500, Florida Gator combo without logo support)
- [ ] `[auto]` Services grid CSS uses asymmetric template, not identical column grid
- [ ] `[auto]` Stats section has dark background (verified by background color parsing)

**Components:**

- [ ] `[auto]` No auto-playing carousels (grep JS for setInterval on carousel init, autoplay attributes on video and audio)
- [ ] `[auto]` No modal popups on page load (grep JS for modal.show, dialog.showModal, alert on DOMContentLoaded or window.load)
- [ ] `[auto]` No "Learn More", "Click Here", "Read More" button text anywhere
- [ ] `[auto]` All icons are Lucide (verified by SVG source path or class name pattern)
- [ ] `[auto]` No client-side framework signatures in output (no React, Vue, Svelte, Next.js, Nuxt, SvelteKit imports)
- [ ] `[auto]` No chatbot widget scripts loaded
- [ ] `[auto]` No custom cursor CSS (`cursor: url(...)` not present)

**Copy:**

- [ ] `[auto]` No "Your Local Experts", "Quality Service You Can Trust", "Your Trusted Partner" or other banned headline patterns
- [ ] `[auto]` No exclamation points in headlines (h1, h2, h3)
- [ ] `[auto]` No all-caps headlines
- [ ] `[auto]` No headlines starting with "We" or "Our"
- [ ] `[auto]` License number appears verbatim if captured (not abstracted to "fully licensed")
- [ ] `[auto]` Owner names appear verbatim if captured (not abstracted to "family-owned")
- [ ] `[auto]` Years-in-business number appears if captured (not abstracted to "years of experience")

**Mobile:**

- [ ] `[auto]` All interactive elements have minimum 44x44px tap target (computed via CSS parsing)
- [ ] `[auto]` Body text minimum 16px on mobile (verified via mobile media query parsing)
- [ ] `[auto]` Mobile sticky header contains phone number as tel: link, visible outside hamburger
- [ ] `[auto]` Fixed bottom elements use `env(safe-area-inset-bottom)` padding
- [ ] `[auto]` No horizontal scroll at 375px viewport width (verified via headless browser render)
- [ ] `[auto]` Every phone number on every page is wrapped in a tel: anchor
- [ ] `[auto]` Form fields have appropriate autocomplete attributes (name, tel, email)

**Dates and placeholders:**

- [ ] `[auto]` Footer copyright year equals current year
- [ ] `[auto]` No placeholder text in output: `[BUSINESS NAME]`, `example.com`, `(555) 555-5555`, `123 Main Street`, `Lorem ipsum`, `John Doe`, `foo@bar`
- [ ] `[auto]` No visible "last updated" dates unless explicitly captured from the prospect
- [ ] `[auto]` All "since [year]" and "[N] years" claims cross-check against `profile.json.business.foundingDate`
- [ ] `[auto]` `dateModified` in schema is within 24 hours of current build time
- [ ] `[auto]` No GRM attribution or "site by" credit in footer

**Trust and fabrication:**

- [ ] `[auto]` Every trust badge, credential, and license number in the HTML matches a value captured in `profile.json.credentials`
- [ ] `[auto]` Every testimonial in the HTML matches a testimonial captured in `profile.json.testimonials`
- [ ] `[auto]` Every service in the HTML matches a service captured in `profile.json.services`

### Human checks (Ron's six-check self-test after Phase 7 deploys)

These run after the preview URL is live. Ron opens the URL on his phone and runs each check in order:

- [ ] `[human]` **Five-second test:** look at the homepage for five seconds. Does it look like a template? Does it look AI-generated? Can I tell the trade without reading? Can I tell the city without reading? Does anything feel off?
- [ ] `[human]` **Swap test:** mentally swap the business name for another contractor in the same trade. Does the site still look fine? If yes, it is too generic.
- [ ] `[human]` **Photography eye test:** are any photos obvious stock? Does the photo set match the trade and the Florida geography?
- [ ] `[human]` **Copy read-aloud test:** read the hero headline and first paragraph out loud. Does it sound like a chatbot wrote it? Does it contain facts that are specific to this one business?
- [ ] `[human]` **Ron text test:** would I text this URL to my sister without a qualifying explanation or an apology?
- [ ] `[human]` **Eye-test gut check:** anything feels off that I can't articulate, investigate what is producing the feeling before shipping

### Any failure blocks the build

Automated check failures block Phase 6 completion and prevent Phase 7 deployment. Human check failures block the handoff to sales and require returning to Phase 5 to regenerate the failing content.

Do not ship slop. "Close enough" is not the standard.
