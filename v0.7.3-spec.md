## Rule 1 — Logo capture: domain filter

### Purpose

Prevent CMS-platform logos (IONOS, Wix, Squarespace, Weebly, GoDaddy, etc.) from being captured as the prospect's logo. v0.7.2 saw IONOS's `logo.gif` served from `cdn.initial-website.com` and (per the audit) correctly rejected it, but the rejection logic was informal. This rule codifies the guard.

### Trigger

During Phase 1 logo capture, when any candidate `<img>` or og:image URL is being evaluated for logo assignment.

### Algorithm

1. Resolve the candidate image URL to absolute form.
2. Extract the hostname.
3. Extract the prospect's domain hostname from the prospect URL.
4. Compare: hostname must either equal the prospect's domain OR be a subdomain of it OR share the registrable domain (e.g., `www.tandfelectric.com` vs `tandfelectric.com` both share `tandfelectric.com`).
5. If the hostname is a known CMS-platform CDN, reject regardless:
   - `cdn.initial-website.com`, `*.initial-website.com` (IONOS)
   - `*.wixstatic.com`, `static.wixstatic.com` (Wix)
   - `*.squarespace-cdn.com`, `images.squarespace-cdn.com` (Squarespace)
   - `*.weebly.com`, `cdn2.editmysite.com` (Weebly)
   - `*.godaddysites.com` (GoDaddy)
6. If rejected, log the rejection with reason `"platform CDN"` and fall through to the next candidate.

### Acceptance criteria

- T&F rebuild: IONOS `logo.gif` at `cdn.initial-website.com/s/img/logo.gif` is rejected with reason `"platform CDN"`.
- A prospect with a real logo at `https://theirdomain.com/images/logo.png` passes the filter.
- A prospect with their logo on a subdomain (`assets.theirdomain.com/logo.png`) passes the filter.

### Test case

Input: `https://www.tandfelectric.com/`
Expected: IONOS platform logo rejected, skill continues to evaluate next candidate (the emotionheader.png, which Rule 2 handles).

### Files affected

- `SKILL.md` Phase 1, step 3 and step 4 — add domain filter language
- `image-handling.md` — add "Logo capture" subsection if one doesn't exist yet (v0.7.2 may have added one; verify)
- New file possible: `references/logo-capture.md` if the logic grows past ~50 lines

## Rule 2 — Logo capture: content-aware solid-color check

### Purpose

v0.7.2's solid-color rejection sampled 9 pixels in a 3×3 grid. If all 9 pixels were within 5 units per channel, the image was rejected as a solid fill. This caught platform background images correctly but produced a false positive on T&F's real logo: the `emotionheader.png` has a red background field that occupies ~60% of the image canvas with the actual logo graphic in the center-left. Random grid sampling missed the logo and the image was thrown out.

The fix: keep the sample check, but add an edge/contour check. Reject only when BOTH the sample is uniform AND the image has negligible edge content.

### Trigger

During Phase 1 logo capture, after Rule 1 (domain filter) passes, before accepting a candidate as the logo asset.

### Algorithm

1. Download the candidate image to a temp path.
2. Run the 9-pixel grid sample check from v0.7.2 (unchanged):
   - Sample at relative positions (0.25, 0.25), (0.5, 0.25), (0.75, 0.25), (0.25, 0.5), (0.5, 0.5), (0.75, 0.5), (0.25, 0.75), (0.5, 0.75), (0.75, 0.75).
   - Compute per-channel delta across all 9 samples.
   - If max delta > 5 per channel, the image is clearly not solid — **accept**, skip the content check, return success.
3. If the sample check says "could be solid," run the content check:
   - Convert image to grayscale.
   - Run edge detection using Pillow's `ImageFilter.FIND_EDGES` (no OpenCV dependency).
   - Count pixels in the edge map with value > 20.
   - Compute edge density = non-zero-count / total-pixels.
   - If edge density > 0.005 (0.5%), the image has real content (logo graphic, text, etc.) — **accept**.
   - If edge density ≤ 0.005, the image is genuinely near-solid — **reject** with reason `"solid fill, no content"`.

### Why 0.005

Empirical threshold. A solid-color JPEG produces edge density near 0 (only compression artifacts). A text logo on a background produces density 0.01–0.03. A complex logo produces 0.05+. 0.005 is a conservative floor that accepts any real logo content while rejecting compression noise.

### Dependencies

Pillow only. Do not add OpenCV — it introduces a native-build dependency that complicates skill portability.

### Acceptance criteria

- T&F rebuild: `emotionheader.png` passes Rule 2 and is accepted as a logo candidate.
- A genuinely solid-red test image (single `#fd040a` fill, no content) is rejected.
- IONOS platform `logo.gif` (already caught by Rule 1) never reaches Rule 2.

### Test case

Input: T&F `emotionheader.png` (940×405, red field with white italic "T&F" + "ELECTRIC" graphic)
Expected: sample check says "uniform-ish red," edge density check finds the logo glyphs, image is accepted.

### Files affected

- Same as Rule 1 — Phase 1 logo capture section in SKILL.md or logo-capture.md

## Rule 3 — Logo capture: crop-to-bounds post-processor

### Purpose

Logo assets often come with heavy background padding. T&F's `emotionheader.png` is 940×405 with the actual logo occupying ~60% of the area centered on a red background with black bars above and below. If used as-is, the nav `<img>` would render the logo at 44px tall but the visible logo inside would be ~26px — undersized and off-center.

The fix: after accepting any logo asset, detect the non-background bounding box and crop to it.

### Trigger

Phase 1 logo capture, after Rule 2 passes (image is accepted), before writing `assets/logo.png`.

### Algorithm

1. Load the accepted image.
2. Identify the background color:
   - Sample the 4 corner pixels.
   - If all 4 are within 10 units per channel, use the average as background color.
   - Otherwise skip Rule 3 entirely — image has no clear padding, use as-is.
3. Build a "content mask": for each pixel, mark as content if it differs from the background color by more than 20 units per channel.
4. Find the bounding box of the content mask (min/max x, min/max y where mask is true).
5. Add a small padding margin (5% of the shorter bounding-box dimension) on each side, clipped to image bounds.
6. Crop the image to this bounding box.
7. Save the cropped version as `assets/logo.png`.
8. Log the crop: original dimensions → new dimensions.

### Edge cases

- If bounding box is smaller than 10% of image area, the content detection likely failed — use the original image, log a warning.
- If bounding box equals the full image, no crop needed — use original.
- If background has two colors (e.g., T&F's black outer + red inner bars), step 2 will detect only the outer black. That's fine — the crop captures the red inner box with the logo inside, which is the desired result.

### Acceptance criteria

- T&F rebuild: `emotionheader.png` (940×405) crops to approximately 720×360 (red box with logo), rejecting the black outer bars.
- A prospect with a tight-fit logo (no padding) comes through unchanged.
- A prospect with gradient background has Rule 3 skipped (step 2 corner check fails).

### Test case

Input: T&F `emotionheader.png` 940×405
Expected: `assets/logo.png` saved at ~720×360 with the red box + logo, outer black padding removed.

### Files affected

- Phase 1 logo capture section
- Possibly a new `scripts/crop_logo_bounds.py` if the logic runs via Python

## Rule 4 — Logo capture: photo signage fallback

### Purpose

Some contractors have no website logo asset at all (or one that all three tiers fail to capture). Before falling to a type-based wordmark (current Tier 3), the skill should scan the top Google Places hero candidate photos for visible signage — vehicle decals, storefront signs, building signage — and offer that as a logo source.

This is not automatic extraction (too error-prone). It's a prompt to Ron during Phase 1 with "I see signage here, should I use this?" and a proposed crop.

### Trigger

Phase 1 logo capture, after Rules 1–3 fail to produce any usable logo asset, before falling to Tier 3 type treatment.

### Algorithm

1. Phase 1 currently runs before Phase 3 photo capture. To access Google Places photos in Phase 1, add a new Phase 1.5 sub-step that pulls just the top 3 Google Places photos (by size + landscape orientation) and caches them at `~/grm-sites-prospects/[slug]/raw-photos/google-preview/`.
2. For each of the 3 photos:
   - Run subregion edge-density sampling (Pillow FIND_EDGES on 10×10 grid cells).
   - Flag any rectangular region with edge density > 0.03 and aspect ratio between 1.5:1 and 4:1 (typical sign proportions) as potential signage.
3. If any photo has a flagged region, prompt Ron:

   ```
   No logo asset found on the prospect's website. I detected possible
   signage in [N] of the top 3 Google Places photos. Preview crops saved to:

     ~/grm-sites-prospects/[slug]/logo-candidates/
       photo-1-signage-crop.png
       photo-2-signage-crop.png

   Options:
   (a) Use photo-1 signage as logo source
   (b) Use photo-2 signage as logo source
   (c) Skip — use type-based wordmark (Tier 3)

   Reply with a, b, or c.
   ```

4. On (a) or (b), save the selected crop as `assets/logo.png`, log the source, continue.
5. On (c) or no signage detected, fall to existing Tier 3 behavior.

### Acceptance criteria

- T&F rebuild: Rule 4 should not fire (Rules 1–3 now capture `emotionheader.png` successfully).
- A synthetic prospect with no website logo and a Google Places photo containing a truck with visible decals: Rule 4 fires, prompts Ron, Ron picks the truck crop, skill continues.
- A prospect with no logo and no photos: Rule 4 finds no candidates, falls to Tier 3 silently.

### Test case

Cannot fully test with T&F on this sprint (Rules 1–3 will capture the real logo first). Validation of Rule 4 requires a deliberate test prospect where the website logo is genuinely absent. Log for next batch of F3–F6 builds — if any prospect hits this path, use that build as the Rule 4 validation case.

### Files affected

- Phase 1 logo capture section — add Rule 4 fallback branch
- New Phase 1.5 sub-step for early Google Places photo fetch
- Possibly a new `scripts/detect_signage.py`

### Risk note

Rule 4 is the most complex in v0.7.3 and the one most likely to produce false positives. Keep it behind a Ron-confirmation prompt (never auto-accept a detected signage crop). If implementation friction is high during Rule 4's commit, defer Rule 4 to v0.7.4 — Rules 1–3 alone fix the T&F regression.

## Rule 5 — OG image: photo composite generator

### Purpose

v0.7.2's OG image generator (Pillow-based, current output: solid red/black graphic with white Helvetica text) fixed the brand-color metric but produced an artifact with zero business context. The v0.7.1 OG image was the raw hero candidate photo (wrong color dominance, right business context).

v0.7.3 generates a composite that carries both: real business photo + brand-color bottom bar with identity text.

### The locked recipe (variant b6, approved 2026-04-13)

1. **Source photo.** The Phase 4-selected primary hero candidate photo, accessed from `assets/photos/[primary-hero-slug]-1600w.jpg` (the largest JPEG size produced by the Phase 3 compression pipeline).
2. **Output dimensions.** 1200 × 630 pixels (OG spec). If source aspect is wider than 1200:630, crop center. If narrower, crop sides to fit.
3. **Top vignette.** Apply a vertical gradient darken to the top 20% of the image (pixels 0 through 20% of height). Gradient: black overlay at 25% opacity at the top edge, ramping linearly to 0% opacity at the 20% mark. This mutes sun flares, sky gradients, or other bright distracting elements without looking obviously filtered.
4. **Bottom bar.** Solid brand-primary color (`profile.json → brandPalette.primary`), 160 pixels tall, pasted at pixel row (height - 160) to (height). Hard crisp top edge — no alpha fade, no gradient.
5. **Text on bar.** All text rendered in white with a dark-primary drop shadow (2px offset, color = darkened primary at ~60% lightness).
   - **Line 1 (tagline):** `profile.json → voice.tagline` if present, else the H1 hero headline. 32px, medium weight (Space Grotesk Medium or Helvetica Neue Medium fallback), rendered 18px from the bar's top edge.
   - **Line 2 (identity):** `[BUSINESS_NAME_UPPERCASE]  |  [PHONE_FORMATTED]`. Example: `T&F ELECTRIC  |  (352) 465-4600`. 38px bold. Two spaces on each side of the pipe. Pipe character rendered at 70% alpha white (not 100%), the rest of the text at 100% white. Business name portion has -1px letter-spacing for typographic density; the pipe and phone stay at default tracking. 12px gap below Line 1.
   - **Line 3 (URL):** prospect's website domain, lowercased, no protocol, no trailing slash. Example: `tandfelectric.com`. 26px regular weight, color `rgb(220, 220, 220)` (muted white), 10px gap below Line 2.
6. **Font.** Primary: Space Grotesk TTF bundled at `~/.claude/skills/prospect-site/fonts/SpaceGrotesk-[weight].ttf` (Regular, Medium, Bold). If the TTF is not present, fall back to Helvetica Neue (macOS system font). Log the font used.
7. **Output path.** `assets/og-image.png`.
8. **Referenced by.** Phase 5.5 meta tag injection, where `<meta property="og:image" content="[canonical URL]/assets/og-image.png">` is written.

### Vertical spacing in the bar (final)

- 18px top pad
- Line 1 (32px text)
- 12px gap
- Line 2 (38px text)
- 10px gap
- Line 3 (26px text)
- 8px bottom pad

Total: 18 + 32 + 12 + 38 + 10 + 26 + 8 = 144px of content in a 160px bar. The remaining 16px sits as the natural top-pad overshoot above Line 1's cap height and the bottom-pad overshoot below Line 3's descender. Centers the block visually without precise vertical-align math.

### Trigger

Phase 5.5, as a new step that runs after Phase 3 has populated the photo manifest and before the meta tag injection step writes OG tags.

### Algorithm

See the recipe above. Implementation is a single Python script using Pillow:

- `scripts/generate_og_composite.py`
- Inputs: path to primary hero photo, path to fonts directory, business name, phone, tagline, URL, brand primary hex, output path
- Outputs: `assets/og-image.png` (1200×630 PNG)

### Acceptance criteria

- T&F rebuild: `assets/og-image.png` matches the b6 mockup visually. Dominant color is brand red. Photo of the T&F building is clearly visible in the upper 73% of the image. Three text lines render in the bottom red bar in the correct order and spacing.
- A prospect with a non-red brand color (e.g., blue): the bar renders in that prospect's brand primary, not red.
- A prospect with no tagline: Line 1 falls back to the H1 hero headline.
- Font fallback path: if Space Grotesk TTF is missing, Helvetica Neue is used and the fallback is logged.

### Test case

Input: T&F build with `assets/photos/tf-building-exterior-1600w.jpg` as primary hero, `brandPalette.primary = "#fd040a"`, `voice.tagline = "Fifty years. Two Tuccis. One phone number."`, business name = "T&F Electric", phone = "(352) 465-4600", URL = "tandfelectric.com".

Expected output: pixel-comparable to the approved b6 mockup at `~/grm-automations/audits/tandf-v07-vs-v072-2026-04-13/og-mockups/variant-b6-final-polish.png`.

### Files affected

- New: `scripts/generate_og_composite.py`
- Updated: `SKILL.md` Phase 5.5 (add composite generation step before meta tag injection)
- Updated: `references/seo-geo.md` OG section (document the composite, reference the script)

## Rule 6 — OG image: entropy gate

### Purpose

Prevent future regressions to flat/template OG graphics by adding an automated gate that rejects OG images without meaningful visual content.

### Trigger

Phase 6, as a new check 18 (inserted after existing check 17).

### Algorithm

1. Load `assets/og-image.png`.
2. Convert to grayscale.
3. Compute the Shannon entropy of the pixel intensity histogram.
4. If entropy < 6.0, fail the build with message: `"OG image entropy [value] below threshold 6.0 — composite lacks visual content. Check Phase 5.5 output and primary hero photo selection."`
5. If entropy ≥ 6.0, pass.

### Why 6.0

- Solid-color image: entropy ~0.
- Two-color graphic (v0.7.2's red/black rectangle): entropy ~2.5–3.5.
- Simple gradient: entropy ~4.0–5.0.
- Photo composite (v0.7.3 b6 recipe): entropy 6.5–7.5.

6.0 is a conservative floor. A real photo composite will clear it easily; a flat graphic will fail hard.

### Acceptance criteria

- T&F rebuild: v0.7.3 b6 composite has entropy 6.5+, passes.
- Simulated regression: manually replace `assets/og-image.png` with v0.7.2's red/black graphic. Phase 6 check 18 fails with entropy ~3.0.

### Test case

Inline in T&F rebuild: confirm Phase 6 passes all 18 checks.

### Files affected

- `SKILL.md` Phase 6 — add check 18 in a new "OG integrity" subsection, update "Phase 6 runs seventeen checks" to "eighteen checks"
- Helper script optional: can be inline Python, doesn't warrant a separate `scripts/` file

## Rule 7 — site-review v1.1: CSS background-image scan

### Purpose

The v1.0 site-review mission's `service-card-images` extractor only reads `<img>` tags, producing a false negative on sites where featured service cards use CSS `background-image` (which is the actual pattern for v0.7.x builds). v0.7.2 featured card rotation is working but site-review reports "no service card images" on both pages.

### Trigger

site-review mission execution, `featured-service-image-rotation` check.

### Algorithm

1. For each page being checked, fetch the rendered HTML.
2. In addition to collecting `<img src>` values for elements matching the service-card class pattern, also parse:
   - Inline `style="background-image: url(...)"` attributes
   - Computed-style `background-image` values via Playwright (grm-browser already uses Playwright, reuse it)
3. Combine both sources into a single list of image URLs per card.
4. Proceed with existing rotation comparison logic (Home vs Services page must use different URLs for equivalent cards).

### Acceptance criteria

- T&F v0.7.3 rebuild: site-review reports `featured-service-image-rotation` PASS with real URL diff (Home uses `gp-09.jpg`, Services uses `gp-05.jpg`), not a "no images detected" false pass.
- A synthetic test site with identical background images on Home and Services cards fails the check (regression test that the extractor is actually reading the URLs, not always passing).

### Files affected

- `~/grm-automations/tools/grm-browser/missions/site-review/` — specific file per grm-browser structure; Claude Code to locate

### Not coupled to skill changes

Rule 7 lives entirely in the site-review mission code. It can be implemented in parallel with any of Rules 1–6 without conflict. Commit it to the grm-automations repo, not the skill repo.
